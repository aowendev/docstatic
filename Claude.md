# docStatic MDX File Format for Tina Editor

This guide provides information for creating docStatic format MDX files that work seamlessly with the Tina CMS editor. docStatic combines Markdown with React components to create rich, interactive documentation.

## Overview

docStatic MDX files are enhanced Markdown files that:
- Include frontmatter metadata for content management
- Support standard Markdown syntax for text content
- Use React components instead of traditional Markdown features like admonitions and footnotes
- Are designed to work with the Tina headless CMS editor
- Should NOT contain code blocks (use CodeSnippet component instead)

## File Structure

### Required Frontmatter

Every MDX file must begin with YAML frontmatter containing metadata:

```yaml
---
lastmod: '2026-01-24T11:43:34.275Z'
title: Your Page Title
description: >-
  A brief description of the content. This can span multiple lines
  and is used for SEO and page previews.
tags:
  - software
  - software_content-management-systems
  - software_content-management-systems_tinacms
  - computer-languages
  - computer-languages_markdown
  - computer-languages_markdown_mdx
draft: false
review: false
translate: false
approved: false
published: true
unlisted: false
---
```

### Frontmatter Fields

- **lastmod**: ISO timestamp of last modification (auto-generated by Tina)
- **title**: Page title (required)
- **description**: Page description for SEO and previews
- **tags**: Array of taxonomic tags for categorization
- **draft**: Whether the content is in draft state
- **review**: Whether the content needs review
- **translate**: Whether the content needs translation
- **approved**: Whether the content has been approved
- **published**: Whether the content is published
- **unlisted**: Whether to hide from navigation

## Content Structure

### Standard Markdown

Use standard Markdown syntax for basic formatting:

```markdown
# Main Heading
## Section Heading
### Subsection Heading

**Bold text** and *italic text*

- Unordered lists
- With multiple items

1. Numbered lists
2. In sequence

[Link text](https://example.com)

Regular paragraphs with line breaks between them.
```

## React Components

docStatic provides specialized React components to replace standard Markdown features and add rich functionality. Use these components with proper JSX syntax.

### Admonitions (Callout Boxes)

Replace Markdown blockquotes with Admonition components:

```jsx
<Admonition type="note" title="Important Note">
  This is a note callout with important information.
</Admonition>

<Admonition type="tip" title="Pro Tip">
  This is a helpful tip for users.
</Admonition>

<Admonition type="caution" title="Be Careful">
  This warns users about potential issues.
</Admonition>

<Admonition type="danger" title="Warning">
  This is for critical warnings.
</Admonition>
```

**Available types**: `note`, `tip`, `info`, `caution`, `danger`

### Figures (Images with Captions)

Use Figure components for images instead of standard Markdown image syntax:

```jsx
<Figure 
  img="/img/screenshot.png" 
  caption="Descriptive caption for the image" 
  size="75" 
  align="center" 
/>
```

**Properties:**
- **img**: Path to image file (required)
- **caption**: Image caption and alt-text (required)
- **size**: Width percentage (e.g., 25, 50, 75, 100)
- **align**: Alignment when size < 100% (`left`, `center`, `right`)

### Comments (Editorial Comments)

Add editorial comments visible only in the CMS:

```jsx
<Comment 
  reviewer="ReviewerName" 
  comment="This section needs more detail about the API parameters" 
/>
```

Comments are invisible in the published site but visible to content editors.

### Footnotes

Use Footnote components instead of Markdown footnote syntax:

```jsx
<Footnote summary="Additional Context">
  Detailed explanation that appears in an expandable section.
</Footnote>
```

### Collapsible Details

Create expandable sections with Details components:

```jsx
<Details summary="Click to expand">
  Content that is initially hidden and can be revealed by clicking
  the summary text. Supports rich content including other components.
</Details>
```

### Code Snippets

For code examples, use CodeSnippet components instead of Markdown code blocks:

```jsx
<CodeSnippet 
  code="console.log('Hello, world!');" 
  language="javascript"
  title="Basic JavaScript Example"
/>
```

### Conditional Text

Show content based on conditions:

```jsx
<ConditionalText conditions="advanced-users">
  This content only appears when the advanced-users condition is met.
</ConditionalText>
```

### Reusable Snippets

Include reusable content snippets:

```jsx
<Snippet snippet="installation-requirements" />
```

### Glossary Terms

Link to glossary definitions:

```jsx
<GlossaryTerm term="static-site-generator">SSG</GlossaryTerm>
```

### Variable Sets

Use dynamic variables in content:

```jsx
<VariableSet variables="product-info">
  The current version is {version} released on {releaseDate}.
</VariableSet>
```

### Tabs

Create tabbed content sections:

```jsx
<Tabs>
  <TabItem value="windows" label="Windows">
    Content specific to Windows users.
  </TabItem>
  <TabItem value="macos" label="macOS">
    Content specific to macOS users.
  </TabItem>
  <TabItem value="linux" label="Linux">
    Content specific to Linux users.
  </TabItem>
</Tabs>
```

### Passthrough (Advanced)

Include raw HTML or JSX for specialized needs:

```jsx
<Passthrough 
  type="html" 
  string="<div class='custom-styling'>Custom HTML content</div>"
  summary="Custom HTML Block"
/>

<Passthrough 
  type="jsx" 
  string="import CustomComponent from '@site/src/components/CustomComponent';"
  summary="Import Statement"
/>
```

## Important Rules and Restrictions

### ❌ Don't Use These Markdown Features

- **Standard admonitions**: `:::note` or `> **Note:**` (use `<Admonition>` component)
- **Markdown footnotes**: `[^1]` syntax (use `<Footnote>` component)  
- **Code blocks for code samples**: Use `<CodeSnippet>` component for actual code examples that may need testing
- **Standard images**: `![alt](src)` for documentation images with captions (use `<Figure>` component instead)

**Note**: Standard Markdown images are still appropriate for:
- Inline SVG icons and UI elements: `![icon](/icons/settings.svg)`
- Pre-scaled images used within text flow
- Simple decorative elements that don't need captions or special styling

**Note**: Standard code blocks are still appropriate for:
- Longer chunks of pre-formatted text that don't need unit testing
- Configuration files, output examples, or non-executable content
- YAML frontmatter examples or documentation formatting

### ✅ Component Best Practices

1. **Use capitalized component names**: `<Figure>` not `<figure>`
2. **Close self-closing tags**: `<Comment />` not `<Comment>`
3. **Quote attribute values**: `type="note"` not `type=note`
4. **No code in content**: Components are for rich content, not code examples
5. **Proper nesting**: Ensure components are properly nested and closed

### Content Guidelines

- Keep content focused on documentation, not code tutorials
- Use semantic HTML structure with proper heading hierarchy
- Include descriptive alt-text in Figure captions for accessibility
- Tag content appropriately for discoverability
- Use Comments for editorial notes that shouldn't appear publicly

## File Organization

Place MDX files in the appropriate directories:

- `/docs/` - Main documentation files
- `/blog/` - Blog posts and announcements  
- `/src/pages/` - Standalone pages
- `/i18n/{locale}/` - Translated content

## Tina CMS Integration

The Tina editor provides:
- Visual editing interface for all components
- Media manager for image uploads
- Rich text editor with component insertion
- Content validation and preview
- Workflow management with draft/review/publish states

Components appear in the Tina editor as insertable blocks with form fields for their properties, making it easy for non-technical users to create rich content without writing JSX manually.
